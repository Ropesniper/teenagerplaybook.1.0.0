🖱 The Pre-Session Agreement Popup

Before any session (Body Postures, Facial Expressions, Intonation) starts, users see a popup asking them to tick a checkbox and click "Begin Session". This popup is defined in js/env-check.js. It replaced a slow automatic camera/mic scanner that took 8+ seconds.

How to change the popup text

Open js/env-check.js. Find the items array near the top of the runEnvironmentCheck() function. There are two lists — one for camera sessions (isCam is true) and one for mic sessions. Edit the strings inside those arrays. Each string is plain text; you can include <strong> tags to bold words.

How to change the consent label text

In the same file, find the <label for="sessionAgreeTick"> HTML inside the template string. Edit that sentence. The camera-specific privacy note ("My camera feed is processed entirely in the browser…") is appended only when isCam is true — controlled by the ${isCam ? '…' : ''} expression.

How to remove the popup entirely

In each session page (bodypostures.html, facialexpression.html, intonation.html), find the runEnvironmentCheck(…, () => { … }) call inside DOMContentLoaded. Replace the whole runEnvironmentCheck(…) call with just the callback body directly. The video will start immediately on page load.

🤖 AI Session Checkpoints — Timing, Instructions & Scoring

Each session video pauses at specific timestamps and activates the camera or microphone for an AI exercise. These pause points are called checkpoints and are fully configurable in the session HTML files.

Where to find the checkpoint config

Open the session file (bodypostures.html, facialexpression.html, or intonation.html) in any text editor. Near the top of the <script> block, you will see a const CHECKPOINTS = […] array. Each entry in the array is one pause point.

What each field means

FieldWhat it doesExample
time Seconds into the video when it pauses for this exercise 30 — pauses at 0:30
replayTime Seconds to jump back to when the user clicks "↩ Replay instructions" 20 — jumps back to 0:20
name Title shown on the exercise overlay panel 'Exercise 2 — Power Pose'
instruction The text instruction shown to the user during the exercise 'Stand with feet wide, hands on hips…'
aiEval Description of what the AI is looking for (shown as a subtitle) 'Wide stance · elbows out · open torso'
evaluator Body Postures only. Which scoring algorithm to use. Options: 'upright', 'widestance', 'chinlevel', 'custom' 'widestance'
emotion Facial Expressions only. Which emotion the AI expects. Options: 'happy', 'neutral', 'sad', 'surprised', 'angry' 'happy'
pattern Intonation only. Which pitch pattern to score. Options: 'rising', 'falling', 'emphasis' 'rising'
minSpeakSeconds Intonation only. Minimum seconds of speech before a score is accepted 2.5
phrase Intonation only. The sentence shown for the user to read aloud '"Are you absolutely sure about that?"'

Example: changing when a checkpoint fires

If your video has a new exercise instruction at 1:20 (80 seconds), change time: 55 to time: 80. If the instruction in the video starts at 1:10 (70 seconds), set replayTime: 70 so users jump back to hear it again.

Example: adding a fourth checkpoint

Copy one of the existing checkpoint objects, paste it after the last one (before the closing ]), and update time, name, instruction, and evaluator/emotion/pattern. The system automatically handles any number of checkpoints.

Changing the pass threshold

Find const PASS_THRESHOLD = 80; in each session file and change the number. 80 means 80% composite AI score required to pass an exercise. Lowering it (e.g. to 65) makes sessions easier; raising it (e.g. to 90) makes them more demanding. This must be changed in each session file separately.

🎬 Uploading & Managing Session Files

All uploaded files live in the /files/ directory on the server. The server accepts any file type — video, PDF, audio, image, ZIP, Office document. Maximum file size is 500 MB per upload.

Standard session videos — required filenames

SessionRequired filenameWhere it's referenced
Body PosturesBodyPostures.mp4bodypostures.html → <source src="files/BodyPostures.mp4">
Facial ExpressionsFacialExpression.mp4facialexpression.html → <source src="files/FacialExpression.mp4">
IntonationIntonation.mp4intonation.html → <source src="files/Intonation.mp4">

These names are case-sensitive. If you upload bodypostures.mp4 (lowercase) the session page will not find it. Use the upload dropdowns which auto-set the correct name.

Personal / custom files

Any file uploaded with a custom name (e.g. Session_James_Week2) is stored in /files/ and can be granted to a specific user via the Notes Inbox reply form. The user then sees a "Begin Session" or "Open" button in their dashboard under "Your Personal Sessions".

How the file library works

Every logged-in user has a 📁 floating button in the bottom-right corner of every page. Clicking it opens the File Library overlay. Admins see all uploaded files and can upload or delete from there. Regular users see only files that have been personally granted to them via note replies.

What happens if a session video is missing

The session page shows a placeholder message: "Video not yet uploaded. Admin Panel → Upload Video → [session name]." The AI exercise overlays will still work — they will just have nothing to pause. Upload the video and the placeholder disappears automatically.

📤 Publishing Files to User Dashboards

The Publish File system lets you push any file from the library directly onto users' dashboards without them needing to request it. Published files appear in a "📂 Published Resources" section that users see at the top of their dashboard.

Visibility options

OptionWho sees the fileUse it when
Everyone All logged-in users, regardless of account type General handouts, session reference sheets, announcement videos for everyone
Specific users Only the individual users you tick in the list Personal session recordings, parent-specific guides, one-to-one materials

How to publish a file

  1. Upload the file first via File Management (above the Publish panel on this page).
  2. Scroll to Publish File. Select the file from the dropdown.
  3. Enter a label (shown to users, e.g. "Week 3 — Posture Reference Sheet").
  4. Optionally add a short description.
  5. Choose visibility: Everyone, or Specific users (tick names from the list).
  6. Click Publish File. The file appears on the chosen users' dashboards immediately.

Difference between Published Files and Personal Files (note grant)

Published Files are proactive — you push them to dashboards without the user needing to ask. They appear under "📂 Published Resources".
Personal Files (note reply grants) are responsive — you grant a file in reply to a user's note request. They appear under "✦ Your Personal Sessions".
Both use files from the same library. You can grant the same file both ways.

Unpublishing a file

In the "Currently Published" list, click Unpublish. The file disappears from dashboards immediately. The file itself remains in the library — it is not deleted.

🐛 Error Reports from Users

Every protected page of the site has a small ⚠ button in the bottom-right corner (above the 📁 file library button). Any logged-in user can click it to report a problem — a broken video, a score that didn't save, a page that won't load, anything. Their report is sent to the server and appears in the Error Reports section on this admin panel.

What you see in each report

FieldWhat it tells you
UsernameWho submitted the report
PageWhich page they were on (e.g. /bodypostures.html)
DescriptionWhat the user described — be as specific as they were
Date/timeWhen the report was submitted

Workflow

  1. Review new (OPEN) reports when you visit the admin panel.
  2. Investigate the issue described. Check the relevant session page yourself.
  3. Click "Mark Resolved" once you've fixed or investigated it — this tags it RESOLVED so you know it's dealt with.
  4. Click "Delete" to remove old resolved reports and keep the inbox clean.

The ⚠ button appears on all pages that load js/auth-guard.js: dashboard, admin, bodypostures, facialexpression, intonation, and history. It is not present on public pages (login, signup, index).

✉️ User Notes & Personal Sessions

Users can send you a note from their dashboard. You reply from this panel. Optionally, you can attach a file from the library to your reply — the user then gets a personalised session card on their dashboard.

Step-by-step: granting a personal session

  1. Upload the file first — File Management section above, or the 📁 library overlay.
  2. Go to User Notes Inbox on this page. Find the user's note.
  3. Write your reply in the "Your Reply" textarea.
  4. In the "File Name" field, type the exact filename including extension (e.g. Session_James.mp4). The hint below the field shows all available files.
  5. Give it a label (e.g. "James — Week 2 Posture Review") and a short description.
  6. Click Send Reply. The user sees the file on their dashboard immediately.

The user's view

When a user has been granted a file, they see a "✦ Your Personal Sessions" section at the top of their dashboard above the standard session cards. Each granted file has an "Open" button that launches the smart file viewer (video player, PDF embed, image view, or download link depending on file type).

Filtering notes by user type

Use the filter chips (All / 🎓 Teenagers / 👨‍👩‍👧 Parents / 📚 Teachers / ✦ Scholars) to see notes from specific user types. The badge next to each username shows which type they are.

📢 Polls & Announcements

Polls and announcements both appear in the "From the Archive Desk" section on every user's dashboard. Announcements are shown as pinned gold banners. Polls are shown as interactive cards with vote buttons.

Creating an announcement

  1. Type your message in the "Question or Announcement Text" box.
  2. Leave all option fields empty.
  3. Click "📌 Publish Announcement".

Announcements show immediately on all logged-in dashboards.

Creating a poll

  1. Type your question in the text box.
  2. Fill in at least 2 option fields. Click "+ Add Option" to add more.
  3. Click "Publish Poll".

Users vote once per poll. They can change their vote by clicking a different option. You can see who voted for what in the admin poll list (voter names shown in brackets). Regular users only see vote counts — they cannot see who voted.

Deleting a poll or announcement

Scroll to "Existing Polls & Announcements" and click Delete next to the item. Deletion is permanent and removes all votes.

👤 User Account Types & Registration

When users sign up, they choose their role. This changes their dashboard experience (greeting, intro text, note prompt) but does not restrict what sessions they can access.

TypeBadgeWho it's forDashboard tone

Parent👨‍👩‍👧 ParentSupporting a childSupportive, home-practice focused
Teacher📚 TeacherClassroom educatorProfessional, cohort-focused
Other✦ ScholarAdult self-learnerNeutral, goal-focused

Camera consent is automatically granted and the camera consent checkbox hidden for Parent and Teacher accounts (since they typically use the platform for observation, not for doing exercises themselves). You can change this in signup.html in the ROLE_META object — set camera: true to require consent for any type.

Removing a user

Registered Users section → Remove button. This permanently deletes the user account, all their session history, and all their notes. It cannot be undone. You cannot remove the admin account.

🖥 Server, Hosting & Health Monitoring

The health banner

When you open this admin panel, the server's /health endpoint is polled immediately and then every 60 seconds. If any issue is detected, a coloured banner appears at the very top of this page (above the nav bar).

Banner colourIconWhat it meansWhat to do
Red / dark red🔴 Critical: database disconnected, or server unreachable Check MongoDB service and Render dashboard. Users cannot log in or save data.
Gold / amber⚠️ Warning: high memory, large file library Restart the server process; or delete unused files from the library.
Blue-ish infoℹ️ Informational: cold start, server just woke up No action needed — dismiss with ✕. This happens when Render wakes from sleep.

Info-level and warning-level banners have a ✕ dismiss button. Critical banners cannot be dismissed — fix the underlying issue first. A blocking popup also appears once per page load for critical issues.

Render free-tier cold starts

On Render's free tier, the server goes to sleep after 15 minutes of inactivity. The first request after sleep takes 20–30 seconds to respond (a "cold start"). This is normal and not an error. To prevent it you can: (a) upgrade to a paid Render plan, or (b) set up a free "ping" service (e.g. UptimeRobot) to request https://your-app.onrender.com/health every 10 minutes — but note that the /health endpoint requires admin auth, so use a public endpoint instead (you can add a GET /ping route in server.js that returns { ok: true } with no middleware).

Environment variables — where secrets live

On Render, go to your service → Environment → Environment Variables. Never put these in any file that goes into Git.

VariableWhat it isHow to change it
PORTPort the server listens onRender sets this automatically; leave it unless you know what you're doing
MONGO_URIMongoDB connection stringGet from MongoDB Atlas → Connect → Drivers
JWT_SECRETSigns all login tokens. Keep it secret and long.Generate with: node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"
ADMIN_USERNAMEYour admin login usernameChange in env vars then restart; the account already in the DB will still work
ADMIN_PASSWORDYour admin passwordChange in env vars then delete your admin user from the DB so it gets re-created on restart
ADMIN_EMAILAdmin account emailSame as above

Restarting the server

On Render: go to your service → top right → "Manual Deploy" → "Deploy latest commit". Or push any commit to your connected repo. On a VPS: pm2 restart server or node server.js in the server/ directory.

MongoDB Atlas — the database

All user accounts, session history, polls, and notes are stored in MongoDB. To view or edit data directly, use MongoDB Compass (desktop app) or the Atlas web UI. Collections: users, histories, polls, notes. The users collection stores hashed passwords — never the plain text. If you need to reset a user's password, delete their document and ask them to re-register, or implement a password reset route in server.js.

➕ Adding a New Session Page (e.g. Parent–Child Session)

  1. Copy an existing session file. Use bodypostures.html as your starting point for camera sessions, or intonation.html for microphone sessions. Rename it (e.g. parent-session.html).
  2. Edit the CHECKPOINTS array. Set your own time values to match your video, update name and instruction fields. Set an appropriate evaluator / emotion / pattern.
  3. Change SESSION_TYPE. Set const SESSION_TYPE = 'parent-session'; (any unique string). Also update const SESSION_KEY to the same string.
  4. Add the session type to server.js HistorySchema. In the History schema, find enum: ['bodypostures', 'facialexpression', 'intonation'] and add your new type.
  5. Add a session card to dashboard.html. Inside the .mirror-grid div, copy one of the three <a class="mirror-card"> blocks, change the href to your new page, and update the icon, title, and description.
  6. Add a video target option. In dashboard.html (admin upload dropdown) and admin.html (File Management dropdown), add an <option value="YourFileName"> entry.
  7. Upload the video using the correct filename via File Management above.

🔒 Security — What's Protected and What to Watch Out For

ProtectionHow it works
PasswordsStored as bcrypt hashes (cost factor 12). The plain-text password is never stored anywhere.
Login tokensJWT signed with JWT_SECRET, valid for 8 days. Stored in localStorage (remember-me) or sessionStorage (tab-only).
Admin routesAll /admin/* and /health routes check the JWT and verify role === 'admin' before responding.
File accessAll files in /files/ are served via an authenticated route. Users without a valid token cannot access them.
Path traversalAll filenames passed to the server are sanitised with path.basename() before being used in file operations. A user cannot read ../server/.env by crafting a filename.
Input limitsNotes capped at 2000 characters. Display names capped at 60. Username validated to letters/numbers/underscore/hyphen only (2–30 chars).
XSSAll user-generated content rendered in innerHTML is passed through an _esc() function that escapes &, <, >, and ".
Self-deletionThe admin account cannot delete itself via the Users panel.
Login rate limitingMaximum 10 failed login attempts per IP address per 15 minutes. After that, the IP is locked out for the remainder of the window.
File accessThe /files/ directory is blocked from static serving. All file access goes through the authenticated /files/:name route.
Keep-alive endpointGET /ping is public and returns {ok:true}. Use this with UptimeRobot to prevent Render cold starts. No sensitive data is exposed.

What you must do before going live: change JWT_SECRET and ADMIN_PASSWORD in your environment variables. The defaults in .env are not safe for production. Generate a strong JWT_SECRET: node -e "console.log(require('crypto').randomBytes(48).toString('hex'))"

What to check periodically: the Registered Users list for unexpected accounts, the Notes Inbox for unusual messages, and the health banner for server issues.
